### 防超賣原理

```go
ts.mu.Lock()              // 1. 加鎖
event := ts.store.GetEvent(id)  // 2. 讀庫存
if event.Remaining < qty { ... } // 3. 判斷
event.Remaining -= qty           // 4. 扣減
ts.store.CreateOrder(order)       // 5. 下單
ts.mu.Unlock()                    // 6. 解鎖
```

步驟 2~5 在同一把 Mutex 內完成，確保任何時刻只有一個 goroutine 執行搶票，
從根本上杜絕了超賣的可能性。

## Go 併發優勢總結

### 1. goroutine 輕量級併發

- 每個 goroutine 初始僅佔用 **2-8 KB** 棧記憶體（對比 Java 線程的 1 MB）
- 本測試中最大同時啟動 **10,000 個 goroutine**，記憶體開銷極低
- Go runtime 的 M:N 調度器將 goroutine 高效分配到 OS 線程上

### 2. sync.Mutex 確保正確性

- 搶票的「讀庫存→判斷→扣減→下單」四步驟在同一把 Mutex 內完成
- 即使 10,000 個 goroutine 同時搶 100 張票，也**絕不超賣**
- 不使用 `defer Unlock()`，手動控制解鎖時機以減少鎖持有時間

### 3. sync/atomic 高效統計

- 計數器（嘗試次數、成功數、失敗數）使用 `atomic.AddInt64`
- 無需加鎖，硬體層級的原子操作，效能遠優於 Mutex
- 適用於「只需要累加/讀取」的場景

### 4. channel + WaitGroup 模式

- **fan-out**: 啟動 N 個 goroutine 併發搶票
- **fan-in**: 透過 buffered channel 收集所有結果
- WaitGroup 確保所有 goroutine 完成後才關閉 channel
- 這是 Go 併發程式設計的經典模式，簡潔且高效

### 5. 兩層鎖分離設計

- **Store 層**: `sync.RWMutex` 保護 map，允許多個讀取同時進行
- **Service 層**: `sync.Mutex` 保護搶票業務邏輯的原子性
- 兩層鎖各司其職，減少不必要的鎖競爭
